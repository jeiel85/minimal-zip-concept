# MZC1 to MZC2 Migration and Compatibility Guide

MZC2는 MZC1(RLE 단독 압축)의 구조적 한계를 극복하고 사전식 하이브리드 압축 체계를 구축하기 위해, 파일 형식 명세의 전면적인 개선을 단행합니다.

본 가이드는 MZC1과 MZC2의 상세한 아키텍처적 차이점 비교 매트릭스, 하위 호환성 유지 전략, 그리고 단계별 순차 구현 로드맵을 제공합니다.

---

## 1. MZC1 vs MZC2 상세 아키텍처 비교 매트릭스

| 비교 항목 (Metrics) | MZC1 포맷 규격 | MZC2 포맷 규격 | 마이그레이션 영향도 및 의의 |
| :--- | :--- | :--- | :--- |
| **Magic Header** | `"MZC1"` (4 bytes) | `"MZC2"` (4 bytes) | 포맷 종류 및 버전을 물리적으로 분별하는 핵심 식별자 변경 |
| **고정 헤더 크기** | **54 bytes** | **56 bytes** | 사전 섹션 범위를 식별하기 위해 `Dictionary Size (2 bytes)`가 삽입됨 |
| **지원 블록 종류** | - `0x00`: Literal Block<br>- `0x01`: Run Block | - `0x00`: Literal Block<br>- `0x01`: Run Block<br>- **`0x02`: Token Block (NEW)** | 사전 엔트리 참조 전용 치환 블록이 페이로드 영역에 추가됨 |
| **최대 알고리즘 모드** | RLE 단독 동작 (`0x01`) | RLE / Dict-Only / **Hybrid (`0x03`)** | 데이터 유형에 맞춰 유연하게 3종의 압축 작동 모드 제어 가능 |
| **에러 핸들링 확장** | 기본 포맷 오류 검증 | **사전 손상 / 토큰 이탈 예외** 검출 | `CorruptDictionary`, `InvalidTokenIndex` 에러 유형 추가로 메모리 안전성 보강 |

## 2. 하위 호환성 및 파일 마이그레이션 아키텍처 전략

MZC2 디코더 도구는 과거에 구 버전 MZC1 포맷으로 압축하여 저장한 백업 파일들 또한 **단일 CLI 배포로 완벽하게 풀 수 있도록 설계 수준에서 하위 호환성(Backward Compatibility)을 보장**해야 합니다.

### 2.1 MZC2 디코더의 이중 파싱 분기 설계

MZC2 Decompress 파이프라인의 입구에서 매직넘버 검사를 수행할 때 다음과 같이 이중 분기(Double Dispatch)를 수행합니다.

```text
       [ 입력 압축 파일 데이터 스트림 로드 ]
                        |
            파일의 시작 4바이트 Magic 분석
           /                            \
     ("MZC1" 검출)                 ("MZC2" 검출)
          |                              |
 MZC1 디코더 파이프라인 실행      헤더 분석 및 Dictionary Size 판독
 (54B 헤더 및 RLE 전용 복원)             |
                                  사전 섹션 & 하이브리드 복원 실행
```

1. **Magic Header == `"MZC1"`**:
   * 즉시 레거시 파싱 분기로 점프하여 54바이트 고정 헤더 규격에 맞춰 `MzcHeader`를 복구합니다.
   * 복원된 데이터를 MZC1 디코더(`rle_decompress`)에 직통 연결하여 복원합니다.
2. **Magic Header == `"MZC2"`**:
   * 최신 56바이트 규격을 사용해 헤더를 파싱하고 `Dictionary Size` 만큼 사전 데이터를 파싱해 로드합니다.
   * `Algorithm Type` 코드에 따라 `0x01`(RLE), `0x02`(Dict), `0x03`(Hybrid)에 특화된 디코더 루프를 안전하게 동작시켜 원본을 복원합니다.

## 3. MZC2 단계별 순차 구현 로드맵 (Development Roadmap)

MZC2는 한 번에 모든 것을 짜기보다는, **가장 작고 검증 가능한 단위**로 나누어 점진적으로 코드를 늘려나가는 테스트 주도형 마일스톤 방식으로 구현할 것을 강력하게 권장합니다.

```text
[ Phase 1 ] ──> [ Phase 2 ] ──> [ Phase 3 ] ──> [ Phase 4 ] ──> [ Phase 5 ] ──> [ Phase 6 ]
데이터 모델      사전 직렬화      패턴 스캐너      인코더 코어      디코더 코어      통합 벤치마크
```

### Phase 1: 데이터 모델 및 구조체 확장 (학습: Struct & Enum)
* `MzcHeader` 구조체를 56바이트 구조로 확장하고, 리틀엔디언 2바이트 사전 크기 직렬화/역직렬화 기능을 설계합니다.
* `MzcError` 열거형에 MZC2 전용 변형(`InvalidTokenIndex`, `CorruptDictionary`)을 추가합니다.

### Phase 2: 가변 사전(Dictionary) 직렬화 엔진 구현 (학습: Byte Array & Iterators)
* 사전을 표현하는 `Dictionary` 구조체를 설계합니다 (`items: Vec<Vec<u8>>` 구조).
* 사전 데이터를 파일에 쓸 수 있도록 `[Count: 2B] + [Length + EntryData]의 연속` 형태로 이진 바이트 스트림으로 패킹하는 로직 및 반대 파싱 로직을 만들고, **사전 단독 이진화 유닛 테스트**를 작성해 검증합니다.

### Phase 3: 가변 슬라이딩 N-gram 스캐너 및 이득 스코어링 모듈 (학습: HashMap & Sorting)
* 입력 데이터 전체를 흐르며 빈도를 분석하는 슬라이딩 윈도우 스캐너 알고리즘을 코딩합니다.
* 점수 공식 `Score = (L - 3) * F - (2 + L)`을 구현해 이득이 높은 순서로 정렬하고 상위 $M$개 단어를 최종 사전으로 리턴하는 유틸리티를 완성합니다.

### Phase 4: 하이브리드 최적 매칭 인코더 구축 (학습: Look-ahead Matching)
* RLE 검출(`run >= 4`?), Longest Match Token 검출, Literal 축적의 우선순위 판정 엔진을 완성합니다.
* 임시 메모리 계산 크기를 비교하여 소형 파일 시 사전 수록을 포기하고 자동으로 RLE 모드로 고쳐 쓰는 적응형 폴백 메커니즘을 완성합니다.

### Phase 5: 하이브리드 디코더 엔진 및 공격 방어 검증 (학습: Boundary Checks)
* 사전 로드 후 페이로드의 세 가지 블록 코드를 순차 복사하는 복원 엔진을 탑재합니다.
* 테스트 플랜의 오류 주입 시나리오(이상한 인덱스 주입, 잘린 사전 데이터 전달)를 인위적으로 동작시키는 통합 테스트를 작성하여, Panic 없이 명확한 에러를 반환하는지 예외 복원망을 완비합니다.

### Phase 6: CLI 인수 확장 및 모드별 벤치마크 기능
* `mzc compress <in> <out> --mode <rle|dict|hybrid>` 아규먼트를 추가합니다.
* 동일 데이터에 대해 RLE 단독, 사전 단독, 하이브리드 압축 시의 크기와 비율 차이를 깔끔한 테이블 형식으로 비교해 출력해 주는 벤치마크 유틸리티를 완성하여 학습 가치를 극대화합니다.
