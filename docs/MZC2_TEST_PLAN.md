# MZC2 Test Plan

MZC2는 가변적 사전 및 하이브리드 복합 블록이 개입되므로, 압축률 증가보다 **프로그램 안정성과 무손실 무결성 확보**를 위한 극단적 예외 주입 테스트를 가장 중요한 철학으로 삼습니다.

본 검증 계획은 MZC2 엔진 구축 시 반드시 수행하고 통과해야 할 핵심 테스트 시나리오들을 규정합니다.

## 2. 예외 및 공격 데이터 주입 테스트 (Tampering & Security Testing)

인코더/디코더는 오염되거나 고의로 훼손된 악성 MZC2 이진 파일을 만났을 때 시스템이 정지(Panic)하거나 메모리 침범 없이 **사전에 지정된 안전한 에러**를 출력해야 합니다.

### 2.1 TC-3.1: 유효 범위를 벗어난 토큰 주입 (Out-of-Bounds Token Index)
* **주입 방법**:
  1. 단어가 딱 5개 등록된 정상적인 MZC2 압축 파일을 생성합니다 (사전 단어 인덱스는 0~4번).
  2. 바이너리 에디터를 이용해 페이로드 영역의 특정 `Token Block (0x02)`의 `Token Index` 필드를 `0x0064` (100번 인덱스)와 같은 엉뚱한 값으로 강제 덮어씁니다.
* **기대 결과**:
  * 디코더는 압축 해제 실행 중, `0x02` 블록의 인덱스 `100`이 사전에 등록된 최대 주소 범위(`4`)를 벗어났음을 즉시 검출해야 합니다.
  * 오류: **`MzcError::InvalidTokenIndex { index: 100, max_valid: 4 }`** 발생 및 실행 중단. (메모리 세그폴트 차단)

### 2.2 TC-3.2: 훼손된 사전 직렬화 레이아웃 (Corrupt Dictionary Layout)
* **주입 방법**:
  1. 헤더의 `Dictionary Size` 필드는 100바이트라고 기록합니다.
  2. 사전 영역 내부의 `Entry Count`는 50개라고 기록하지만, 실제 이진 바이트는 단 10바이트만 배치하여 파일을 고의로 잘라(Truncate) 보냅니다.
* **기대 결과**:
  * 사전 파싱 엔진은 100바이트를 채 읽기 전에 데이터 스트림이 종결되었음을 감지해야 합니다.
  * 오류: **`MzcError::CorruptDictionary`** 혹은 **`MzcError::TruncatedHeader`** 계열의 디코드 실패 에러 반환.

## 3. 압축 알고리즘 모드별 벤치마크 및 비교 검증 (TC-4.0)

MZC2 CLI 도구는 사용자에게 다양한 압축 전략의 효율을 한눈에 체득할 수 있도록 모드 전환을 제공하며, 각 모드 간의 상호 복구와 파일 호환성(Cross-Compatibility)이 유기적으로 검증되어야 합니다.

```bash
# 1. RLE 단독 압축 모드 수행
./target/release/mzc compress --mode rle input.txt output.mzc
# 2. 사전 전용 압축 모드 수행
./target/release/mzc compress --mode dict input.txt output.mzc
# 3. 하이브리드 자동 압축 모드 수행
./target/release/mzc compress --mode hybrid input.txt output.mzc
```

* **교차 작동성 검증:**
  * 어떤 모드로 압축되었든 상관없이, 헤더의 `Algorithm Type` (`0x01` / `0x02` / `0x03`) 필드를 보고 디코더는 **스스로 최적의 파이프라인을 자동 인식하여 Decompress 처리를 완료**할 수 있어야 합니다. (사용자가 복원 시에는 `--mode` 옵션을 줄 필요가 없게 설계)
