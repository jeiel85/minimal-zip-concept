# MZC2 Algorithm Design

MZC2는 단순한 RLE(Run-Length Encoding)를 뛰어넘어 가변 바이트 슬라이스 패턴을 수집하고 치환하는 **사전식 압축(Dictionary-based Compression)**과 MZC1의 장점을 결합한 **하이브리드(Hybrid) 압축 알고리즘**을 도입합니다.

본 설계서는 알고리즘의 원리를 수학적, 논리적, 그리고 Rust 구현 친화적 관점에서 상술합니다.

---

## 1. 사전 구축 단계 (Dictionary Construction Process)

사전식 압축의 핵심은 **"어떤 바이트 패턴을 사전에 등록할 것인가?"**를 결정하는 패턴 선정 전략입니다. 텍스트와 이진 바이너리를 모두 수용하기 위해, 단어 단위 쪼개기가 아닌 **가변 슬라이딩 N-gram 스캔 기술**을 적용합니다.

```text
       슬라이딩 윈도우 (N-gram 스캔)
[ H e l l o _ W o r l d _ H e l l o ]
  ├───────┤
    "Hello" 패턴 감지 (빈도수 = 2)
```

### 1.1 패턴 이득 평가 공식 (Score Evaluation Function)
토큰 블록(`0x02`)은 파일 내에 `Type (1바이트) + Index (2바이트) = 3바이트`로 저장됩니다.
따라서, 사전에 등록할 패턴의 크기 $L$이 3바이트 이하라면 치환 시 오히려 파일 크기가 증가하거나 동등하여 압축의 이득이 전혀 없습니다. 

패턴의 이득을 평가하기 위해 아래의 **평가 함수(Evaluation Score)**를 설계합니다.

$$\text{Score}(P) = (L - 3) \times F - (2 + L)$$

* **$L$ (Length):** 바이트 패턴 $P$의 크기 (바이트 수, 반드시 $L > 3$ 이어야 유의미)
* **$F$ (Frequency):** 파일 전체에서 해당 패턴 $P$가 중복 출현한 총 빈도수
* **$-(2 + L)$:** 이 패턴을 사전 섹션에 등록하기 위한 직렬화 오버헤드 패널티 (`Entry Length (1 byte) + Entry Data (L bytes) + Entry Count Overhead 분할 분배`)

### 1.2 사전 추출 파이프라인
1. **바이트 스캔**: 입력 바이트 배열 전체를 슬라이딩 윈도우 방식으로 훑어 길이가 $4 \le L \le 32$ 사이인 모든 N-gram 바이트 시퀀스의 빈도를 계산합니다.
2. **이득 산출**: 중복 검출된 각 시퀀스 후보군의 `Score`를 위 공식에 대입해 계산합니다.
3. **사전 정렬 및 상위 수록**: `Score`가 0보다 높은 후보 중 가장 높은 점수를 가진 상위 $M$개(최대 65,535개)를 최종 사전 엔트리로 선정하여 사전을 생성합니다.

## 2. 하이브리드 최적 블록 판정 매칭 알고리즘

압축 단계에서 인코더는 입력 바이트 배열의 매 위치 $i$마다 RLE(런), 사전 토큰, 일반 리터럴 중 **가장 크기 감소율이 높은 최적의 블록을 탐색하여 방출**하는 탐욕적(Greedy) 매칭 전략을 채택합니다.

```text
                      [ 입력 데이터 스트림 ]
                                |
             +------------------+------------------+
             |                                     |
   [ RLE 런 판정 (>= 4?) ]                 [ 사전 매칭 판정 ]
             |                                     |
       (참) -> Run Block                     (참) -> Token Block
             |                                     |
             +------------------+------------------+
                                |
                           (모두 실패)
                                |
                       -> Literal Block 버퍼링
```

### 2.1 블록 선택 우선순위 및 규칙
매 포인터 위치 $i$에서 다음과 같이 우선순위를 판정합니다:

1. **RLE 런 블록(`0x01`) 최우선 고려**:
   * 만약 `data[i]`가 4회 이상 연속 반복된다면, RLE 압축의 효율이 가장 압도적이므로 이를 우선하여 `Run Block`으로 즉시 인코딩해 출력합니다.
2. **사전 토큰 블록(`0x02`) 2순위 고려 (최장 일치 매칭 - Longest Match)**:
   * 연속된 동일 문자가 아니지만, 사전 수록 엔트리 중 현재 `data[i..]` 위치에서 시작해 일치하는 가장 긴 단어(Longest Match)를 탐색합니다.
   * 일치하는 엔트리 인덱스 주소가 존재할 경우, `Token Block`으로 방출하고 포인터를 일치한 바이트 길이만큼 스킵 전진시킵니다.
3. **리터럴 블록(`0x00`) 최종 폴백**:
   * 위 두 조건에 모두 부합하지 못하면, 현재 1바이트를 `Literal Buffer`에 대기 적재하고 포인터를 1칸 우측 전진시킵니다. (대기 적재량이 65,535바이트가 되거나 다음 런/토큰 매칭이 발견되면 하나의 리터럴 블록으로 묶어 방출합니다.)

## 3. 작은 파일 및 사전 오버헤드 대응 방안: 적응형 폴백 (Adaptive Fallback)

사전 기반 압축은 **"사전 자체를 파일 내부에 직렬화하여 송신한다"**는 점에서 크기가 매우 작거나(예: 1KB 미만) 반복 패턴의 종류가 파편화된 파일에서는 사전 직렬화 크기 오버헤드 때문에 **압축 파일이 원본보다 훨씬 커지는 현상**이 심화됩니다.

MZC2는 이를 방어하기 위해 **적응형 모드 폴백 알고리즘(Adaptive Mode Switching)**을 탑재합니다.

```text
                  [ 인메모리 임시 압축 수행 ]
                                |
             +------------------+------------------+
             |                                     |
    (사전 포함 하이브리드 크기)              (MZC1 RLE 전용 크기)
             |                                     |
             +------------------+------------------+
                                |
                    [ 압축 효율 비교 연산 ]
                                |
            사전 오버헤드가 더 큰가? OR 원본보다 큰가?
                   /                           \
               (Yes)                           (No)
                 |                               |
       MZC1 (RLE-only) 혹은             MZC2 하이브리드로 저장
       Raw(No compression)로 폴백
```

* **비교 판단 프로세스:**
  1. 인코더는 최종 디스크 쓰기 직전, 메모리 버퍼 상에서 `사전 데이터 + 압축 페이로드`의 크기 $S_{\text{hybrid}}$와 MZC1 방식의 크기 $S_{\text{rle}}$, 그리고 원본 크기 $S_{\text{orig}}$를 시뮬레이션 비교합니다.
  2. 만약 $S_{\text{orig}} \le S_{\text{hybrid}}$ 이거나 $S_{\text{rle}} < S_{\text{hybrid}}$ 일 경우, 인코더는 사전을 빌드하지 않고 사전 크기(Dictionary Size)를 `0`으로 리셋합니다.
  3. 헤더의 `Algorithm Type`을 `0x01` (RLE Only) 모드로 고쳐서 디스크에 쓰고, 본문은 오직 RLE 블록들과 리터럴 블록만으로 구성합니다. (0바이트 사전 헤더)

## 4. 디코딩 역직렬화 및 무손실 복원 파이프라인

압축 해제기는 한 방향 스캔만으로 완벽하고 빠른 속도로 오버플로우와 침범 없이 무손실 원상 복구를 완수합니다.

```text
[ MZC2 이진 데이터 스트림 ]
      |
      +---> 1. 고정 56바이트 헤더 디코딩 및 매직넘버 검사
      +---> 2. Dictionary Size 바이트만큼 사전 영역 분리 -> 메모리 인덱스 맵 로드
      +---> 3. 나머지 Payload 영역을 바이트 블록 루프 디코딩:
                 - Type 0x00 (Literal) -> Raw 바이트 복사
                 - Type 0x01 (Run)     -> 값 반복 복사
                 - Type 0x02 (Token)   -> 사전 인덱스 참조 복사
      +---> 4. 최종 데이터의 SHA-256 해시 검산 후 원본 해시와 대조
```

사전의 각 엔트리 주소와 복원된 데이터 버퍼를 순서대로 결합하므로, 원본 바이너리와 단 1비트의 어긋남 없이 완벽하게 복원이 보장됩니다.
